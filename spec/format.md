# HFP Format

> **Draft / pre-alpha.**

An `.hfp` file is a valid HTML document. It opens in any browser, and its data is
machine-readable without rendering. Machine-readable blocks live in the `<head>` and are
identified by stable `id` attributes.

## Blocks

| Block | `id` / type | Purpose |
|-------|-------------|---------|
| Version meta | `<meta name="hfp-version">` | Format version (e.g. `1.0`) |
| Template id | `<meta name="hfp-id">` | UUID of the template; signed by the author |
| Author meta | `<meta name="hfp-author">` | Author identification |
| Schema | `#hfp-schema`, `application/json` | Field definitions (JSON Schema subset) |
| Data | `#hfp-data`, `application/json` | Filled-in data, readable without rendering |
| Author signature | `#hfp-author-signature`, `application/pkcs7-signature` | CMS detached signature over the canonical document |
| Data signature | `#hfp-data-signature`, `application/pkcs7-signature` | CMS detached signature over the canonical data |
| Attachment | `application/hfp-blob` script, `id="hfp-blob-<id>"` | Binary attachment as Base64 |

## Identity

- **`hfp-id`** — template id, generated by the author at sign time, part of the author
  signature.
- **`instance-id`** — UUID of a single filled instance, assigned by the host on first
  open and written into `#hfp-data`.

Deduplication in downstream systems uses `instance-id`; template audit trails use `hfp-id`.

## Schema (JSON Schema subset)

v1 supports a deliberate subset: types, `required`, `enum`, regex, arrays, and blob-refs.
The schema is a **data contract only** — it constrains the shape of the data, never the UX.
Fields may be marked `readonly: true` (pre-filled and locked by the author, part of the
author signature).

## Attachments (blob-ref)

Binary attachments are stored as Base64 inside `application/hfp-blob` script blocks. In the
JSON data they are referenced as a string: `"photo": "blob:hfp-blob-a3f1c"`. A processor
resolves the reference via `getElementById` and decodes the Base64.

## Unknown blocks and versioning

Unknown blocks are ignored for interpretation but **preserved** through canonicalization
(removing them would break signatures). Reserved for future use: `#hfp-processing`
(document type, contract version, idempotency key for batch dedup).
